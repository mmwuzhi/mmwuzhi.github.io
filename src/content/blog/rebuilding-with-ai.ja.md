ここ数日、vibe codingを試してみたくて、Claude Codeを使ってこのサイトを一から作った。

「AIにコードを書かせる」よりもっと手前から始めた。サイトのコンセプト、機能、名前——全部AIと話し合いながら決めた。最終的にAstro + 素のCSS、三言語対応（中国語・英語・日本語）、Blog・Garden・Nowというセクション構成に落ち着いた。名前の*Lacuna*は「欠如」や「空白」を意味するラテン語で、AIが出してくれた候補の中で一番気に入ったものだ。

機能がひととおり揃ったころ、いろんなものが壊れていることに気づいた。カテゴリフィルターをクリックしても何も起きない、テーマの切り替えボタンが反応しない、ページ間を移動するとダークモードがリセットされる、Gardenのタグフィルターも同じく機能しない。調べてみると根本的な設計ミスだった。完全な静的サイトなのに、一部のロジックがSSRの前提で書かれていて、ビルド時にデータが固定されてしまい、実行時に意味をなさなかった。

バグを修正する過程は思ったより面白かった。これまで気にしていなかったことをいくつか学んだ。

## AstroのView Transitionsとスクリプトの挙動

このサイトはAstroのView Transitionsを使っている。ページ間のナビゲーション時に完全なリロードをせず、DOMの一部を入れ替える仕組みだ。これが微妙な問題を生んだ。ダークモード制御のためにJavaScriptで`<html>`に`data-theme`属性を付けていたのだが、ナビゲーションのたびに消えてしまう——AstroがHTML要素ごと置き換えていたからだ。

解決策は`astro:before-swap`イベントを監視し、スワップの直前に古いドキュメントから新しいドキュメントへ属性をコピーすること：

```js
document.addEventListener('astro:before-swap', (e) => {
  const theme = document.documentElement.getAttribute('data-theme');
  if (theme) e.newDocument.documentElement.setAttribute('data-theme', theme);
});
```

関連する問題もあった。Astroの`<script>`タグはデフォルトでESモジュールとして扱われ、セッション中に一度しか実行されない。その中でイベントリスナーをボタンに紐付けても、次のナビゲーションでDOMが入れ替わると消えてしまう。正しいやり方は`astro:page-load`を監視して毎回再登録すること。フレームワークの設計であってバグではないが、気づきにくい。

さらに見つけにくいのが、`<script is:inline>`をトップレベルのlayoutコンポーネントの外に書くと、`</html>`タグの後ろに配置されてしまうという挙動だ。ブラウザは実行してくれるが、View Transitionsのページ入れ替え時には再実行されない。検索機能のデバッグで何度も詰まって、やっとこれが原因だとわかった。

## Scoped CSSの限界

AstroのコンポーネントCSSはローカルスコープで、セレクターに自動で`data-astro-cid-*`属性が付加される。問題は次のように書いたとき：

```css
[data-theme='dark'] .icon-light { display: none; }
```

Astroはこれを次のように変換する：

```css
[data-theme='dark'][data-astro-cid-xxx] .icon-light[data-astro-cid-xxx] { ... }
```

しかし`data-theme`は`<html>`に付いていて、`<html>`にはスコープ属性がない。だからルールは永遠にマッチしない。`:global([data-theme='dark'])`と書けば解決する。初めて踏んだときは原因が全くわからなかった。

## AIとコードを書くということ

AIに頼るとプログラマーとして劣化するという意見がある。自分で考えなくなるからというのがその理由だ。少なくとも今回の経験ではそうは感じなかった。デバッグの際は依然として何が問題かを理解する必要があったし、出力されたコードがまともかどうか判断するのも自分だった。AIはむしろ「いつでも呼べる相棒」に近い感覚で、方向性と判断は自分が持ち、調査と実装はAIに任せるという形だった。

もちろんAIが自信満々に間違いを犯すこともある。`/* @vite-ignore */`のケースがそうだった。解決できないダイナミックインポートに対するViteの警告を抑制するためのコメントだったが、ビルド時にファイルが存在しないためRollupが「エラー」（警告ではなく）を出していた。AIは両者の違いを見落としていた。こういうときはやはり自分でもちゃんと確認する必要がある。

まとめると、今回でモバイルのハンバーガーメニュー、全文検索（Pagefind）、長い記事への目次を追加できた。バグ修正と合わせて、ようやく使えるものになった感じがする。

サイト自体が目的ではない。
